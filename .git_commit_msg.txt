Fix analytics build-time embedding and migrate/seed interaction bug

Found by finally bringing up ALL 6 services (postgres, migrate, cms,
storefront, umami, caddy) together, including a real reverse-proxy test
through Caddy — the full stack had never actually been run together
before this.

1. VITE_ANALYTICS_URL/VITE_UMAMI_WEBSITE_ID were only wired into
   docker-compose.yml's runtime `environment:` block. Vite embeds
   VITE_* vars into the client bundle at BUILD time — as written, the
   tracking script would never have rendered regardless of runtime
   values. Added them as proper build `args` in both
   apps/storefront/Dockerfile and docker-compose.yml.

2. The `migrate` service never set NODE_ENV=production, so Payload's
   runtime defaulted to dev-mode behavior for any command run through
   it — including the seed script. Seeding silently left the database
   in a state that made a LATER `payload migrate` call detect "dev mode
   drift" and hang forever on an interactive y/N prompt that can never
   be answered in a non-TTY container. Reproduced precisely (migrate ->
   seed -> migrate hangs), fixed by setting NODE_ENV=production on the
   migrate service, confirmed on a fresh database that the exact same
   sequence now completes cleanly every time.

Verified end-to-end against a fresh database with all 6 services
running together: real Umami website created via its API, storefront
built with the real website ID baked into the compiled bundle
(confirmed via the actual rendered <script> tag), and all three Caddy
routes (storefront/cms/analytics, via Host-header-based routing over
plain HTTP to isolate the reverse-proxy logic itself from real-DNS/ACME
concerns, which were already understood to need real deployment) proxy
correctly to their backends with real content confirmed, not just
status codes. Both pnpm typecheck passes (storefront, cms) stayed
clean.
